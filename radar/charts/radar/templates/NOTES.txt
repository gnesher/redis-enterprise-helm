{{- if eq .Values.deployMode "cloud" }}

WARNING: no credential KEK is configured (credentials.kek / credentials.existingSecret).
In cloud mode the KEK must be DELIVERED — it is never generated — so the API and
worker pods will FAIL CLOSED (CrashLoopBackOff with "KEK must be provided") and
will not serve traffic until a KEK is present. This is intentional: cloud runs
multiple replicas, and letting each generate its own key would silently render
credentials sealed by one replica undecryptable by another. Provide a stable
32-byte raw KEK before the pods can start:
  head -c 32 /dev/urandom > kek.bin
  kubectl create secret generic radar-credentials --from-file=CREDENTIAL_KEK=./kek.bin
  # then set credentials.existingSecret=radar-credentials

Do not base64 the value before setting credentials.kek. The chart base64-encodes
that literal value into the Secret, so inline credentials.kek must already be
exactly 32 raw bytes and is dev-only.
{{- else }}

WARNING: no credential KEK is configured (credentials.kek / credentials.existingSecret).
Credential encryption is always on, so each pod will generate its OWN ephemeral
KEK at {{ .Values.credentials.kekPath }} on the container filesystem. Credentials
sealed by one pod become PERMANENTLY UNDECRYPTABLE after a restart or on any other
replica. Before storing real connector/auth credentials, provide a stable 32-byte
raw KEK:
  head -c 32 /dev/urandom > kek.bin
  kubectl create secret generic radar-credentials --from-file=CREDENTIAL_KEK=./kek.bin
  # then set credentials.existingSecret=radar-credentials

Inline credentials.kek is dev-only and must be exactly 32 raw bytes, not base64.
{{- end }}
